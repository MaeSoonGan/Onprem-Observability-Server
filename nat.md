# 임시 NAT용 pfSense 구성

## 1. 개요 및 프로젝트 내 역할

- pfSense NAT VM은 MaeSoonGan Observability 서버 구축 과정에서 임시 인터넷 연결을 제공하기 위해 구성한 NAT 장비입니다.

- 본 프로젝트의 Monitoring VM들은 내부 전용망인 10.6.10.0/24 대역에 위치합니다. 이 내부망은 외부 인터넷과 직접 연결되지 않기 때문에, Ubuntu 패키지 설치, Docker 설치, Grafana 관련 저장소 접근, 각종 설치 파일 다운로드를 위해 별도의 NAT 경로가 필요했습니다.

- pfSense는 외부 연결용 가상 스위치와 Monitoring 내부망 사이에 배치되어, 내부 VM들이 외부 인터넷으로 나갈 수 있도록 NAT 역할을 수행했습니다. 또한 로컬 PC 또는 외부 관리망에서 pfSense GUI에 접근할 수 있도록 VyOS Router와 연계한 라우팅 설정도 함께 구성했습니다.

- 이 VM은 운영 서비스용 영구 구성이라기보다, 초기 구축 단계에서 패키지 설치와 외부 저장소 접근을 지원하기 위한 임시 NAT 장비로 사용했습니다.

</br>

## 2. 구성 환경 및 사양

| 구분              | 내용                      |
| --------------- | ----------------------- |
| VM              | pfSense NAT VM          |
| 역할              | 임시 NAT, 패키지 설치용 인터넷 연결  |
| Hypervisor      | VMware ESXi             |
| OS              | pfSense                 |
| vCPU            | 1 vCPU                  |
| RAM             | 1~2GB                   |
| Disk            | 8GB                     |
| Network Adapter | 2개                      |
| WAN Interface   | vmx0                    |
| LAN Interface   | vmx1                    |
| WAN IP 방식       | DHCP                    |
| LAN IP          | 10.6.10.2/24            |
| VyOS Router IP  | 10.6.10.1               |
| 관리 GUI          | pfSense WebConfigurator |

- pfSense NAT VM은 WAN과 LAN을 분리하기 위해 네트워크 어댑터를 2개 사용했습니다. WAN 인터페이스는 외부 연결용 가상 스위치에 연결하고, LAN 인터페이스는 Monitoring 내부망에 연결했습니다.

| Adapter | 연결 대상                | 역할                  |
| ------- | -------------------- | ------------------- |
| vmx0    | 외부 연결용 가상 스위치        | WAN, DHCP 기반 외부망 연결 |
| vmx1    | Monitoring 내부 가상 스위치 | LAN, 10.6.10.2/24   |

</br>

## 3. 선택 이유 및 대안 비교

### 선택 이유

- pfSense를 선택한 이유는 GUI 기반으로 NAT, 방화벽, 라우팅 설정을 확인하고 수정하기 쉽기 때문입니다.

- 본 프로젝트에서는 Monitoring VM들이 초기 패키지 설치를 위해 외부 인터넷에 접근해야 했습니다. pfSense는 WAN을 DHCP로 받아오고, LAN에 내부 IP를 지정한 뒤 NAT를 제공하는 구성이 비교적 단순합니다. 또한 WebConfigurator를 통해 방화벽 Rule, Gateway, Static Route를 시각적으로 확인할 수 있어 실습 환경에서 빠르게 구성하고 문제를 파악하기 적합했습니다.

### 대안 비교

| 대안        | 장점                                 | 한계                                         | 최종 판단              |
| --------- | ---------------------------------- | ------------------------------------------ | ------------------ |
| pfSense   | GUI 기반으로 NAT, 방화벽, 라우팅 설정이 직관적입니다. | 라우팅 중심 문서화는 VyOS보다 명령어 재현성이 낮습니다.          | 임시 NAT 용도로 선택했습니다. |
| VyOS NAT  | CLI 기반으로 라우팅과 NAT를 함께 관리할 수 있습니다.  | NAT 설정까지 VyOS에 집중되면 라우터 역할과 NAT 역할이 섞입니다.  | 라우팅 역할로 분리했습니다.    |
| Linux NAT | Ubuntu 등으로 NAT 구성이 가능합니다.          | iptables, forwarding, DNS 설정을 직접 조합해야 합니다. | 제외했습니다.            |
| 외부망 직접 연결 | 구성이 단순합니다.                         | 내부망과 외부망 분리가 약해지고 구조 설명이 어려워집니다.           | 제외했습니다.            |

</br>

## 4. 설치 및 주요 설정

### 4-1. 포트그룹 및 VM 생성

- 초기 패키지 설치를 위해 Monitoring 내부망에서 외부 인터넷으로 나갈 수 있는 NAT 경로가 필요했습니다.

- 외부 연결용 가상 스위치는 이미 구성되어 있었기 때문에, 해당 스위치에 pfSense WAN 연결용 포트그룹을 생성했습니다. 이후 pfSense VM을 생성하고, 네트워크 어댑터를 2개 연결했습니다.

| Adapter | 연결 대상          | 설명                    |
| ------- | -------------- | --------------------- |
| WAN     | 외부 연결용 가상 스위치  | DHCP로 외부망 IP를 할당받습니다. |
| LAN     | Monitoring 내부망 | 10.6.10.2/24를 할당합니다.  |

### 4-2. pfSense 초기 인터페이스 설정

- pfSense 초기 설정에서는 VLAN을 사용하지 않았습니다. 따라서 VLAN 설정 여부를 묻는 단계에서 `n`을 입력했습니다.

- 이후 MAC 주소를 확인하여 WAN과 LAN 인터페이스를 구분했습니다.

| Interface | 용도  | 설정           |
| --------- | --- | ------------ |
| vmx0      | WAN | DHCP         |
| vmx1      | LAN | 10.6.10.2/24 |

- LAN IPv4 설정 값은 다음과 같습니다.

| 항목                                            | 값         |
| --------------------------------------------- | --------- |
| Configure IPv4 address LAN interface via DHCP | n         |
| LAN IPv4 Address                              | 10.6.10.2 |
| Subnet Bit Count                              | 24        |
| LAN Upstream Gateway                          | Enter     |
| Configure IPv6 via DHCP6                      | n         |
| IPv6 Address                                  | Enter     |
| Enable DHCP Server on LAN                     | n         |
| Revert to HTTP webConfigurator                | n         |

- LAN에 DHCP Server를 켜지 않은 이유는 Monitoring VM들이 고정 IP를 사용하기 때문입니다. 각 VM은 10.6.10.0/24 대역에서 수동으로 IP를 설정하고, Gateway는 VyOS Router인 10.6.10.1을 사용합니다.

### 4-3. GUI 초기 설정

- pfSense 초기 설정 이후 WebConfigurator에 접속하여 GUI 기반 설정을 진행했습니다.

| 항목        | 내용              |
| --------- | --------------- |
| 기본 ID     | admin           |
| 기본 PW     | pfSense         |
| GUI 접근 방식 | WebConfigurator |

- 보안을 위해 실제 운영 환경에서는 초기 비밀번호를 반드시 변경해야 합니다. 본 프로젝트에서는 실습 환경에서 초기 설정을 빠르게 완료한 뒤 필요한 라우팅과 방화벽 설정을 진행했습니다.

### 4-4. VyOS Router 연계 Gateway 설정

- 로컬 PC 또는 10.3.0.0 대역에서 pfSense GUI에 접근할 수 있도록 VyOS Router와 연결되는 Gateway를 생성했습니다.

| 항목         | 값                    |
| ---------- | -------------------- |
| Gateway 대상 | VyOS Router          |
| Gateway IP | 10.6.10.1            |
| 목적         | 10.3.0.0 대역 응답 경로 설정 |

- pfSense 입장에서는 10.3.0.0 대역이 직접 연결된 네트워크가 아니기 때문에, 해당 대역으로 응답을 보내려면 10.6.10.1인 VyOS Router를 통해 응답하도록 라우팅 설정이 필요했습니다.

### 4-5. Static Route 설정

- 10.3.0.0 대역에서 pfSense로 들어온 요청에 대해 응답 경로를 지정하기 위해 Static Route를 설정했습니다.

| 목적지 대역      | Gateway   | 설명                    |
| ----------- | --------- | --------------------- |
| 10.3.0.0/24 | 10.6.10.1 | 로컬 또는 Aruba 연결망 응답 경로 |

이 설정을 통해 pfSense는 10.3.0.0 대역으로 향하는 응답을 VyOS Router로 전달할 수 있습니다.

### 4-6. 방화벽 Rule 설정

- pfSense는 기본적으로 방화벽 정책을 통해 외부망에서 들어오는 트래픽을 제한합니다. 따라서 10.3.0.0 대역에서 pfSense GUI에 접근하거나 필요한 관리 통신을 수행하려면 방화벽 Rule을 설정해야 합니다.

- 설정 방향은 다음과 같습니다.

| 항목          | 내용                             |
| ----------- | ------------------------------ |
| Source      | 10.3.0.0/24                    |
| Destination | pfSense LAN IP 또는 필요한 대상       |
| Action      | Pass                           |
| 목적          | 10.3.0.0 대역에서 pfSense 관리 접근 허용 |

- 실습 중에는 초기 접근을 위해 `pfctl -d` 명령어로 방화벽을 임시 비활성화했지만, 최종 구성에서는 필요한 Source와 Destination에 대해서만 Rule을 추가하는 방식이 더 적절합니다.

</br>

## 5. 동작 흐름 및 트러블슈팅

### 동작 흐름

1. Monitoring VM이 패키지 설치 또는 외부 저장소 접근을 위해 외부 요청을 보냅니다.
2. Monitoring VM의 Gateway인 10.6.10.1로 요청이 전달됩니다.
3. VyOS Router는 기본 라우트에 따라 요청을 pfSense NAT VM인 10.6.10.2로 전달합니다.
4. pfSense는 LAN에서 들어온 요청을 WAN 인터페이스로 NAT 처리하여 외부망으로 전달합니다.
5. 외부망에서 응답이 돌아오면 pfSense가 다시 내부망으로 전달합니다.
6. 응답은 VyOS Router를 거쳐 Monitoring VM으로 돌아옵니다.
7. Monitoring VM은 필요한 패키지 설치와 외부 저장소 접근을 수행합니다.

### 트러블슈팅

#### 문제 1. 10.3.0.x 대역에서 pfSense에 접근되지 않는 문제

| 구분    | 내용                                                                                    |
| ----- | ------------------------------------------------------------------------------------- |
| 현상    | 10.3.0.x 대역에서 다른 10.6.10.x VM들은 접근되었지만, pfSense에는 접근되지 않았습니다.                         |
| 원인    | pfSense 자체 방화벽이 외부 대역에서 들어오는 접근을 차단하고 있었습니다.                                          |
| 해결 과정 | 초기 설정을 위해 `pfctl -d` 명령어로 방화벽을 임시 비활성화한 뒤 GUI 접근을 시도했습니다.                             |
| 결과    | pfSense GUI에 접근할 수 있었고, 이후 필요한 방화벽 Rule을 설정할 수 있었습니다.                                 |
| 재발 방지 | 운영 단계에서는 방화벽을 전체 비활성화하지 않고, 필요한 Source 대역과 Destination에 대해서만 Pass Rule을 추가하도록 정리했습니다. |

#### 문제 2. 방화벽을 내려도 10.3.0.x 대역으로 응답이 가지 않는 문제

| 구분    | 내용                                                                      |
| ----- | ----------------------------------------------------------------------- |
| 현상    | pfSense 방화벽을 임시 비활성화해도 10.3.0.x 대역에서 pfSense GUI 접속이 안정적으로 되지 않았습니다.    |
| 원인    | pfSense가 10.3.0.0 대역으로 응답을 보내기 위한 라우팅 경로를 알지 못했습니다.                     |
| 해결 과정 | 10.3.0.0 대역으로 향하는 응답을 VyOS Router인 10.6.10.1로 보내도록 라우팅을 추가했습니다.         |
| 결과    | 10.3.0.x 대역에서 pfSense GUI 접근이 가능해졌습니다.                                  |
| 재발 방지 | pfSense가 직접 연결하지 않은 네트워크 대역에 응답해야 하는 경우 Static Route를 먼저 확인하도록 절차화했습니다. |

#### 문제 3. 임시 라우팅이 재부팅 후 사라지는 문제

| 구분    | 내용                                                                              |
| ----- | ------------------------------------------------------------------------------- |
| 현상    | `route add`로 임시 라우팅을 추가한 뒤에는 접속이 되었지만, pfSense 재부팅 이후 GUI 연결이 다시 끊겼습니다.         |
| 원인    | CLI에서 추가한 임시 라우팅은 재부팅 후 유지되지 않았습니다.                                             |
| 해결 과정 | 다시 임시 라우팅을 추가하여 GUI에 접근한 뒤, WebConfigurator에서 Gateway와 Static Route를 영구 설정했습니다. |
| 결과    | 재부팅 이후에도 10.3.0.0 대역에서 pfSense로 접근할 수 있는 경로가 유지되었습니다.                           |
| 재발 방지 | 임시 명령어로 확인한 설정은 반드시 GUI 또는 설정 파일을 통해 영구 설정으로 반영하도록 정리했습니다.                      |

#### 문제 4. Monitoring VM에서 인터넷 접근이 되지 않는 문제

| 구분    | 내용                                                                                                       |
| ----- | -------------------------------------------------------------------------------------------------------- |
| 현상    | Monitoring VM에서 패키지 설치를 위한 외부 인터넷 접근이 되지 않았습니다.                                                          |
| 원인    | VyOS Router의 기본 라우트가 pfSense NAT VM으로 향하지 않거나, pfSense WAN이 DHCP로 정상 IP를 받지 못했을 가능성이 있었습니다.              |
| 해결 과정 | VyOS Router에서 `0.0.0.0/0` 기본 라우트가 10.6.10.2로 설정되어 있는지 확인하고, pfSense WAN 인터페이스가 DHCP로 IP를 받은 상태인지 확인했습니다. |
| 결과    | Monitoring VM에서 pfSense NAT를 통해 외부 저장소와 패키지 서버에 접근할 수 있었습니다.                                             |
| 재발 방지 | NAT 경로 점검 시 Monitoring VM Gateway, VyOS 기본 라우트, pfSense LAN IP, pfSense WAN DHCP 상태를 순서대로 확인하도록 정리했습니다.  |
